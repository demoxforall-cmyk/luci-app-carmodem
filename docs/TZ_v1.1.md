## 1. Назначение

Самостоятельный модуль (набор пакетов) для ImmortalWRT, встраивающийся в LuCI, который объединяет:
- **полный набор управляющих функций QModem** (band-lock, выбор RAT, SIM, TTL, AT-консоль, SMS), адаптированный под конкретное железо;
- **мониторинг и UX из видео-демонстрации заказчика** (дашборд телеметрии, юстировка антенн, спидтест, USSD), реализованные собственной адаптивной вёрсткой.

Область: **только мониторинг и управление модемом/соединением**. Вне области: QoS/SQM, борьба с bufferbloat, watchdog задержек, bridge/passthrough, GNSS, eSIM (см. раздел 9).

## 3. Архитектура

### 3.1 Бэкенд соединения — ModemManager
Управление соединением и параметрами — через **ModemManager (`mmcli`)**. WWAN-интерфейс создаётся и управляется штатно через **`proto 'modemmanager'`** (пакет `luci-proto-modemmanager` как зависимость). Дозвон/реконнект — средствами netifd (`ifup`/`ifdown`); ModemManager — исполнитель. Режим — **NAT/router**, без bridge.

**Владение конфигурацией:** модуль создаёт и редактирует WWAN-секцию в `/etc/config/network` и является единственным управляющим модемом. При первой установке выполняется **резервная копия текущего `network`-конфига** и показ вносимых изменений (возможность отката).

### 3.2 Источники телеметрии — MM-first, AT только для глубоких метрик [v1.1, проверено на железе]
Граница между ModemManager и AT уточнена и **подтверждена на живом RM520N-GL** (Banana Pi R4). Принцип: брать максимум из ModemManager, AT-слой использовать только там, где MM не покрывает.

**Берётся из ModemManager (AT не требуется):**
- идентификация (производитель/модель/прошивка/IMEI/номер телефона), оператор/PLMN, регистрация, состояние сессии, access tech — `mmcli -m 0`;
- сигнал агрегированный RSRP/RSRQ/RSSI/SINR (и 5G NR, когда есть) — `--signal-setup` + `--signal-get`;
- RAT (allowed/preferred) и грубый band-lock — Modes / `--set-current-bands`;
- SIM-слоты, IP/DNS/шлюз/трафик сессии — `mmcli -m 0` / bearer `mmcli -b 0`;
- SMS (список/отправка/удаление, длинные+кириллица) — `--messaging-*`;
- PIN/PUK — `-i 0 --pin=/--puk=`;
- Cell ID / LAC — `--location-get-3gpp`;
- **соседние соты (PCI/EARFCN/RSRP/RSRQ)** — `--get-cell-info` (метод `GetCellInfo`);
- скан операторов — `--3gpp-scan`; событийный мониторинг — `--monitor-state`; сброс — `--reset`/`--factory-reset`.

**Берётся только через Quectel AT (MM не покрывает, подтверждено):**
- по-антенные значения Rx0–3 — `AT+QRSRP` / `AT+QRSRQ` / `AT+QSINR`;
- компоненты carrier aggregation (PCC/SCC) — `AT+QCAINFO`;
- Timing Advance — `AT+QENG="servingcell"`;
- температура модема — `AT+QTEMP`;
- тонкий покабельный NR band-lock — `AT+QNWPREFCFG`.

Полная карта «параметр → источник» — приложение Б.

Нюансы для парсеров (проверено): `GetCellInfo` отдаёт PCI в **hex** (у `AT+QENG` — decimal); TA и RSSI/SINR соседей в `GetCellInfo` отсутствуют (TA только из `AT+QENG`); по-антенное значение `-32768` = «нет тракта» → рендерится как «N/A»; 5G-поля появляются только при 5G-регистрации.

«Плоскость данных» (дозвон через MM) и «плоскость телеметрии» (AT) разделены. Поскольку сигнал и соседние соты теперь идут через MM (по каналу cdc-wdm0), **конкуренция за ttyUSB снижена** — AT-порт нужен лишь для пяти команд выше.

### 3.3 AT-демон (сериализация)
Единый ubus/rpcd-демон с **очередью команд** — единственная точка доступа к AT-порту (`ttyUSB2`/`ttyUSB3`). Нужен для пяти «глубоких» команд (QRSRP/QRSRQ/QSINR, QCAINFO, QENG servingcell→TA, QTEMP, QNWPREFCFG), ручной AT-консоли и USSD. Исключает гонки между этими потребителями и защищает от обрывов/«мусора» в движении. **Обязательный компонент** (нагрузка на него снижена за счёт переноса сигнала и соседних сот на MM, см. 3.2).

### 3.4 Опрос
Опрашивается **только активный экран**. Интервалы: **Setup antenna — 1 с**, остальные экраны — **2–3 с**. Команды телеметрии — лёгкие, идут через демон по очереди. Влияния на джиттер/задержку пользовательского трафика нет (AT — управляющий канал, отдельный от канала данных).

### 3.5 RPC и хранилище
ubus-объект `carmodem` (rpcd ACL): чтение статуса/телеметрии, управление band/RAT/SIM/TTL, дозвон, SMS/USSD, запуск спидтеста. Конфиг — UCI `/etc/config/carmodem`. История метрик для realtime-графиков — кольцевой буфер в tmpfs/памяти.

### 3.6 Состав пакетов
- `luci-app-carmodem` — UI всех экранов меню «Modem»;
- `carmodem-core` — бэкенд: AT-демон, ubus/RPC, парсеры телеметрии, интеграция с MM;
- зависимости: `luci-proto-modemmanager`, `modemmanager`, `librespeed-cli` (или аналог), стандартные luci-base/rpcd.

## 4. Функциональные требования — управление

### 4.1 Расширенная панель модема
- FR-1. Выбор интерфейса модема (`/dev/cdc-wdm*`) и связанного индекса ModemManager; показ привязок (`cdc-wdm ↔ mmcli N`).
- FR-2. Выбор SIM-слота (SIM1/SIM2).
- FR-3. Задание входящего и исходящего TTL.
- FR-4. Выбор режима **RAT**: пресеты (2G/3G/4G/5G и комбинации с Pref-приоритетом); индикация текущего режима. Предупреждение о возможных ограничениях RAT через ModemManager.
- FR-5. **Band-lock**: сетки LTE (B1…B71), 5G NR (n1…n79), 3G/2G; «Включить все частоты» / «Выключить все частоты».
- FR-6. «Сохранить и применить» с предупреждением о возможном кратковременном обрыве связи; применение «на лету»; UI-подтверждение действия.

### 4.2 Соединение
- FR-7. Включение/выключение передачи данных; индикация статуса сессии.
- FR-8. Профиль: APN/APN2, тип PDP (IPv4/IPv6/IPv4v6), аутентификация (NONE/PAP/CHAP/PAP-CHAP), PIN, метрика, DNS — через `proto modemmanager`.
- FR-9. Авто-реконнект средствами netifd; перезагрузка/сброс модема (с подтверждением).

### 4.3 Отладка
- FR-10. AT-консоль (произвольные команды + ответ) — через AT-демон.
- FR-11. Кастомные AT на этапах post-init / pre-dial с задержками.

## 5. Функциональные требования — мониторинг и UX

### 5.1 Экран «Modem Info» (главный дашборд)
- FR-12. **Сеть и подключение**: PLMN (напр. 25001 MTS RUS), статус/технология (`connected (lte/nr)`), время сессии, IP WWAN (IPv4/IPv6), тип (приём/отдача), общий трафик, **ссылка «показать соту на карте»** (deep-link 4cells.ru по MCC/MNC/eNB/CID).
- FR-13. Параметры соты: Global Cell ID, ID базовой станции (eNB), ID соты (CID), кол-во ячеек, MCC, MNC, TAC, PCI, Timing Advance. [Cell ID/TAC — MM location-3gpp; eNB+сектор — разбор hex Cell ID; TA — `AT+QENG`]
- FR-14. Ведущий диапазон (LTE/NR): band + EARFCN/NR-ARFCN + ширина полосы (BW).
- FR-15. Качество сигнала: RSRP, RSRQ, SINR, RSSI — с цветовой индикацией (раздел 7).
- FR-16. **По-антенные значения** (Antenna 0–3): RSRP/RSRQ/RSSI/SINR.
- FR-17. **Мощность передатчика** (TX power).
- FR-18. **Агрегация частот (CA)**: компоненты PCC/SCC — band, EARFCN/ARFCN, PCI, RSRP, состояние (activated/deactivated).
- FR-19. **5G NSA/NR**: NR band, NR-ARFCN, NR RSRP/SINR.
- FR-20. **Скорость передачи** в реальном времени (↓/↑) + график.
- FR-21. **Соседние сектора**: таблица PCI/EARFCN/RSRP/RSRQ. Источник — **MM `--get-cell-info`** (PCI приходит в hex; RSSI/SINR соседей и TA отсутствуют). [v1.1: перенесено с AT на MM]
- FR-22. **Информация об устройстве** (раскрытие): производитель, модель, аппаратная ревизия, версия прошивки, IMEI, IMSI, номер телефона (с маскированием).
- FR-23. **Информация о роутере**.

### 5.2 Экран «Setup antenna» (юстировка)
- FR-24. Крупноформатные цветные индикаторы **RSRP / RSRQ / SINR** для юстировки антенн в реальном времени.
- FR-25. Шапка: порт модема, период обновления (по умолч. 1 с), «Обновить сейчас», оператор, технология, PCI, сигнал (band/EARFCN/BW), индикатор «LIVE SYNC».
- FR-26. Авто-обновление 1 с; устойчивость к движению.

### 5.3 Экран «Internet Status» (скорость/спидтест)
- FR-27. **График скорости** в реальном времени (↓/↑, скользящее окно ~1 мин).
- FR-28. Сводка: средняя (за выборку) ↓/↑ и пиковая (за сессию) ↓/↑.
- FR-29. **Полный спидтест на LibreSpeed CLI** (публичные серверы): ↓, ↑, ping, jitter; результат в JSON. Перед запуском — предупреждение «тест нагрузит канал».
- FR-30. Лёгкий запасной тест: загрузка тестового файла по URL (быстрая проверка канала).

### 5.4 Экран «SMS-managment»
- FR-31. Приём/просмотр/история SMS; отправка (текст + длинные/кириллица). Реализация через **MM messaging-API** (`--messaging-*`), который сам кодирует UCS2 и собирает длинные сообщения; AT-PDU — резерв. Экспорт в ФС. Пересылка — вне области. [v1.1: источник — MM]

### 5.5 Экран «USSD-команды»
- FR-32. Отправка USSD и отображение ответа (баланс/сервисные коды) с **декодированием кириллицы (UCS2)**; сессия сериализована с фоновым опросом.

### 5.6 Сканирование операторов [v1.1, новое в объёме]
- FR-32a. Кнопка «Сканировать сети» — `mmcli -m 0 --3gpp-scan` со списком доступных операторов (PLMN, имя, технология, статус available/current/forbidden). Перед запуском — предупреждение: скан занимает время и кратко влияет на связь.

### 5.7 Событийный мониторинг состояния [v1.1, новое]
- FR-32b. Отслеживание смены состояния/технологии/регистрации событийно (`--monitor-state` / D-Bus signals) — мгновенная реакция на потерю регистрации или переключение LTE↔5G без поллинга.

### 5.6 Карта (внешняя интеграция)
- FR-33. Кнопка/ссылка открытия внешнего сервиса карт покрытия (4cells.ru или аналог) с передачей текущих идентификаторов соты. Сам картсервис не разрабатывается.

## 6. Обработка отсутствующих данных
- FR-34. Если метрика недоступна на конкретном RM520N (модем не отдаёт): элемент остаётся на месте и показывает **«N/A» / прочерк**, не скрывается и не вызывает ошибку.

## 7. Цветовые пороги качества (стандартные, зашитые)
Эталон для UI и тест-кейсов (точные числа подтвердить при ревизии):

| Метрика | Зелёный (отл.) | Жёлто-зел. (хор.) | Жёлтый/оранж. (средне) | Красный (плохо) |
|---|---|---|---|---|
| RSRP, dBm | ≥ −80 | −80…−90 | −90…−100 | < −100 |
| RSRQ, dB | ≥ −10 | −10…−15 | −15…−20 | < −20 |
| SINR, dB | ≥ 20 | 13…20 | 0…13 | < 0 |
| RSSI, dBm | ≥ −65 | −65…−85 | −85…−95 | < −95 |

## 13. Риски
- R-1. Полнота телеметрии RM520N зависит от поддержки `AT+Q...` его прошивкой — проверить на устройстве (часть полей может быть «N/A»).
- R-2. Часть RAT-режимов может работать нестабильно через ModemManager (отмечено и в ролике) — фиксируем ограничение.
- R-3. Доступность/формат deep-link внешней карты (4cells.ru) вне нашего контроля.
- R-4. Спидтест на публичных серверах LibreSpeed зависит от наличия близкого узла и сетевых условий.

## Приложение А. Журнал решений (18)
1. Бэкенд соединения — **ModemManager (mmcli)**.
2. Владение — **модуль создаёт и рулит WWAN** через `proto modemmanager` (+ бэкап network при установке).
3. Режим WWAN — **NAT/router, без bridge**.
4. Низкая задержка/QoS/watchdog — **вне области** (только мониторинг+управление).
5. GNSS — **нет**; карта по текущей соте (deep-link).
6. eSIM — **убран**.
7. USSD — **есть** (декодирование кириллицы, сериализация).
8. SMS — **приём/отправка/история** (длинные/кириллица), без пересылки.

## Приложение Б. Карта источников «параметр → MM или AT» [v1.1, проверено на железе]

**ModemManager (AT не нужен):** идентификация/IMEI/номер; оператор/PLMN/регистрация/состояние/access tech; сигнал RSRP/RSRQ/RSSI/SINR (`--signal-setup`); 5G NR сигнал; RAT и грубый band-lock; SIM-слоты; IP/DNS/трафик (bearer); SMS (`--messaging-*`); PIN/PUK; Cell ID/LAC (`--location-get-3gpp`); соседние соты (`--get-cell-info`); скан операторов (`--3gpp-scan`); событийный мониторинг (`--monitor-state`); сброс (`--reset`/`--factory-reset`).

**Только AT (через AT-демон):** по-антенные Rx0–3 (`AT+QRSRP/QRSRQ/QSINR`); CA-компоненты (`AT+QCAINFO`); Timing Advance (`AT+QENG="servingcell"`); температура (`AT+QTEMP`); тонкий NR band-lock (`AT+QNWPREFCFG`).

## Приложение В. Факты по целевому модему (сняты с RM520N-GL)
- Прошивка: `RM520NGLAAR01A06M4G` (`AT+QGMR`: `..._01.001.01.001`).
- Порты MM: `cdc-wdm0` (qmi, primary), `ttyUSB1` (gps), `ttyUSB2`/`ttyUSB3` (at), `ttyUSB0` (ignored), `wwan0` (net). Рабочий AT-порт — **ttyUSB2/3**.
- Поддержка: gsm-umts, lte, 5gnr; IP: ipv4/ipv6/ipv4v6.
- SIM: slot1 active, **slot2 none**; статус блокировки — **sim-pin2 / fixed-dialing (FDN)** — показывать в статусе.
- По-антенные: реально 2 тракта (Rx0/Rx1), Rx2/Rx3 = `-32768` → «N/A».
- `GetCellInfo`: PCI в hex; serving+neighbour отдаёт; без TA/RSSI/SINR соседей.
- 5G-поля = N/A при LTE-регистрации (модем на сети MegaFon, LTE Band 7 на момент снятия).
- GNSS присутствует (`ttyUSB1`, location gps-raw/nmea) — **вне области**, не используем.

## Приложение Г. Артефакты для этапа тест-кейсов
Сняты реальные AT-ответы (фикстуры): `servingcell` (LTE), `neighbourcell`, `QCAINFO`, `QRSRP/QRSRQ/QSINR`, `QTEMP`, `QNWINFO`, `QSPN`, `QNWPREFCFG`, плюс выводы `mmcli -m 0`, `--signal-get`, `--get-cell-info`. Прототип парсера `servingcell` валидирован (hex Cell ID → eNB+сектор, TAC, BW). Это основа mock-модема и юнит-тестов парсеров (тесты без железа).
